💡 POR QUE SEPARAR A CONFIG?

Centralizar parâmetros em um único notebook de configuração é uma boa prática de engenharia: qualquer mudança de ticker, chave de API ou caminho é feita em um único lugar, e todos os notebooks que executam %run Config/config automaticamente herdam as novas configurações — eliminando duplicação e reduzindo risco de inconsistências.

Bronze/ingestao_api_alpha_spark — Ingestão + PySpark

Bronze/ingestao_api_alpha_spark PYTHON · PYSPARK · DATABRICKS 
# ──────────────────────────────────────────────────────────────────
# Bronze · Ingestão de Cotações B3 via Alpha Vantage → PySpark → Delta
# ──────────────────────────────────────────────────────────────────

# Importar configurações centralizadas
%run /ProjetoAPI_cotacoes/Config/config

import requests
import pandas as pd
from datetime import datetime
from pyspark.sql import SparkSession

# Obter sessão Spark ativa (já disponível no Databricks)
spark = SparkSession.builder.getOrCreate()


def buscar_dados_acao(ticker_b3: str, api_key: str) -> pd.DataFrame | None:
    """
    Consulta a Alpha Vantage API e retorna um DataFrame Pandas
    com as cotações diárias do ticker informado.

    Parâmetros:
        ticker_b3: Código da ação na B3 (ex: 'PETR4')
        api_key:   Chave da API Alpha Vantage

    Retorna:
        DataFrame com colunas: data, abertura, alta, baixa,
        fechamento, volume, ticker, data_ingestao
        Retorna None em caso de erro.
    """
    # Adicionar sufixo .SA para ações da B3
    ticker = ticker_b3 + ".SA"

    # Montar requisição para o endpoint TIME_SERIES_DAILY
    params = {
        "function":   "TIME_SERIES_DAILY",
        "symbol":     ticker,
        "apikey":     api_key,
        "outputsize": "compact"  # últimos 100 dias
    }
    response = requests.get("https://www.alphavantage.co/query", params=params)

    # Tratar erros HTTP
    if response.status_code != 200:
        print(f"[{ticker_b3}] Erro HTTP {response.status_code}")
        return None

    data = response.json()

    # Validar resposta da API (ex: rate limit atingido)
    if "Time Series (Daily)" not in data:
        print(f"[{ticker_b3}] Sem dados. Verifique o rate limit ou a chave.")
        return None

    # Transformar JSON em DataFrame Pandas
    df = pd.DataFrame.from_dict(
        data["Time Series (Daily)"], orient="index"
    )
    df.columns = ["abertura", "alta", "baixa", "fechamento", "volume"]
    df              = df.astype(float)
    df.index        = pd.to_datetime(df.index)
    df              = df.sort_index()  # ordem cronológica
    df["ticker"]       = ticker_b3
    df["data_ingestao"] = datetime.now()

    return df.reset_index().rename(columns={"index": "data"})


# ── Coletar dados para todos os tickers configurados ──────────────
dfs = []

for ticker in tickers:              # variável do config
    print(f"[+] Buscando {ticker}...")
    df = buscar_dados_acao(ticker, API_KEY)
    if df is not None:
        dfs.append(df)

# ── Concatenar e converter para Spark DataFrame ───────────────────
if dfs:
    df_final = pd.concat(dfs, ignore_index=True)
    df_spark = spark.createDataFrame(df_final)

    # ── Persistir em Delta Lake (append para acumular histórico) ──
    df_spark.write \
        .format("delta") \
        .mode("append") \
        .save(bronze_path)        # caminho do config

    print(f"✅ {len(df_final)} registros salvos com sucesso!")
else:
    print("⚠️  Nenhum dado retornado. Verifique API Key e rate limits.")
🔶 POR QUE APPEND MODE NO DELTA?

O modo "append" permite acumular o histórico a cada nova execução sem sobrescrever dados anteriores. O Delta Lake garante a consistência com transações ACID — se a escrita falhar no meio, o estado anterior é preservado automaticamente. Para forçar uma recarga completa, basta trocar para "overwrite".

Visualizacao/view_dados_spark — Análise com Spark SQL

Visualizacao/view_dados_spark PYTHON · SPARK SQL · DATABRICKS 
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()

# ── 1. Leitura da tabela Bronze via catálogo ──────────────────────
df_cotacoes = spark.table("spark_catalog.bronze.cotacoes")

# Registrar como view temporária para uso em células SQL
df_cotacoes.createOrReplaceTempView("cotacoes")

# ── 2. Resumo estatístico por ação ────────────────────────────────
spark.sql("""
    SELECT
        ticker,
        MAX(data)         AS ultima_data,
        MIN(data)         AS primeira_data,
        ROUND(AVG(fechamento), 2) AS media_fechamento,
        ROUND(MAX(fechamento), 2) AS max_fechamento,
        ROUND(MIN(fechamento), 2) AS min_fechamento,
        COUNT(*)          AS total_registros
    FROM cotacoes
    GROUP BY ticker
    ORDER BY media_fechamento DESC
""").show(truncate=False)

# ── 3. Últimas 5 cotações de cada ação ────────────────────────────
spark.sql("""
    SELECT *
    FROM (
        SELECT *,
               ROW_NUMBER() OVER (PARTITION BY ticker ORDER BY data DESC) AS rn
        FROM cotacoes
    )
    WHERE rn <= 5
    ORDER BY ticker, data DESC
""").show(25, truncate=False)

# ── 4. Variação diária (%) ────────────────────────────────────────
spark.sql("""
    SELECT
        ticker,
        data,
        fechamento,
        LAG(fechamento) OVER (PARTITION BY ticker ORDER BY data) AS fechamento_anterior,
        ROUND(
            (fechamento - LAG(fechamento) OVER (PARTITION BY ticker ORDER BY data))
            / LAG(fechamento) OVER (PARTITION BY ticker ORDER BY data) * 100, 2
        ) AS variacao_pct
    FROM cotacoes
    ORDER BY ticker, data DESC
    LIMIT 50
""").show(truncate=False)
📐 SPARK SQL vs DATAFRAME API

Ambas as abordagens são equivalentes em performance — o Spark compila as duas para o mesmo plano de execução físico (Catalyst Optimizer). O SQL é preferido para análises exploratórias por ser mais legível, enquanto a DataFrame API é mais adequada para pipelines programáticos com lógica condicional e reuso de código.

🧊 DELTA LAKE — VANTAGENS SOBRE PARQUET SIMPLES

O formato Delta Lake adiciona ao Parquet: (1) ACID transactions — escrita atômica, sem dados corrompidos; (2) Time Travel — VERSION AS OF N para consultar versões anteriores; (3) Schema Evolution — adição de colunas sem reescrever a tabela; (4) OPTIMIZE + ZORDER — compactação de arquivos pequenos para melhor performance.

Benefícios Pedagógicos & Técnicos

🔍
Transparência do Processo
Cada etapa é visível e explícita: da requisição HTTP ao arquivo Delta. Não há "magia" — alunos compreendem exatamente onde os dados estão e como foram transformados em cada passo.
⚡
Escalabilidade com Big Data
A conversão de Pandas para PySpark demonstra como o mesmo código se adapta de um único laptop a um cluster com centenas de nós — preparando para workloads de nível produção.
🔶
Persistência Inteligente
Delta Lake introduz conceitos de Data Engineering moderno: ACID, time travel, schema evolution e compactação — habilidades diretamente aplicáveis em ambientes cloud reais (AWS, Azure, GCP).
🏗️
Padrões de Mercado
A separação em Config + Bronze + Visualização reflete a Medallion Architecture usada por empresas como Databricks, Netflix e LinkedIn — construindo intuição para arquiteturas de Data Lake enterprise.
🌐
Contexto Real (B3)
Trabalhar com dados reais de ações brasileiras (PETR4, VALE3, ITUB4…) cria engajamento imediato e permite validação intuitiva — alunos podem verificar os dados no dia a dia do mercado.
🔄
Pipeline Evolutivo
O projeto serve como base para evoluções: adicionar camada Silver (limpeza, normalização), Gold (métricas de mercado), alertas automáticos, ou migração para AWS Glue + Redshift.
🚀 PRÓXIMOS PASSOS SUGERIDOS

Silver Layer: Normalização de valores, tratamento de splits e proventos, cálculo de retornos diários e médias móveis (MA20, MA50) via PySpark Window Functions.

Gold Layer: Métricas agregadas por ticker — Sharpe Ratio, Volatilidade Histórica, Máximo Drawdown — prontas para dashboards executivos.

Orquestração: Agendamento diário com Apache Airflow ou Databricks Workflows para execução automática após o fechamento do pregão às 17h.
